# 提交、评测轮询与断点续做

## 运行测试并读取结果

流程：切到「测试用例」tab → 等约 1.5s → 点「运行测试」→ 等约 10s → 读输出。
`[class*="answerInput"]` 内有两个只读 CodeMirror：[0]=实际输出、[1]=预期输出。
`match = (实际 === 预期)`，是"直接提交"还是"喂给 AI 调试"的分岔依据。

若页面没返回 output，切到「编译器输出」tab 取错误文本（点"运行测试"会触发一次编译），
把编译错误喂给模型 debug。

## 提交后轮询评测结果

点「提交本题作答」后弹出评测 modal，**必须轮询到最终态**才判成败，不能急着下结论。

- 中间态（继续等）：`等待评测`、`评测中`、`排队中`
- 最终态：`答案正确`、`答案错误`、`编译错误`、`运行超时`、`段错误`、
  `内存超限`、`输出超限`、`格式错误`、`部分正确`

轮询要点：
- 每约 1s 探测一次，最长约 180s（`SUBMIT_RESULT_TIMEOUT_MS`）。
- modal 选择器用双写法兼容 hash 后缀：`.modal_I0D4Y, [class*="pc-modal"]`；
  用 `modal.offsetHeight === 0` 判断是否可见（display:none 时为 0）。
- 状态文本正则：`/状态\s*：?\s*(\S+)/`。
- 拿到最终态后解析细节：
  - 分数 `/分数\s*：?\s*(\d+)\s*\/\s*(\d+)/`
  - 用时 `/用时\s*：?\s*(\d+)\s*\/\s*(\d+)\s*ms/`
  - 内存 `/内存\s*：?\s*(\d+)\s*\/\s*(\d+)\s*KB/`
  - 评测详情表 `modal table tbody tr`，每行 td：id / 内存 / 时间 / 结果 / 得分。
- 读完自动点 modal 关闭按钮并等约 500ms，避免遮挡下一步操作。
- `passed = (status === '答案正确')`。

## 两级重试策略

区分"业务失败"与"临时网络故障"，不要混在一起：

| 级别 | 阈值（默认） | 触发条件 |
|---|---|---|
| 业务重试 | 5 | 测试不匹配 / 提交未过 → 带错误详情喂给模型重生成 |
| 网络重试 | 3 | 502 / 503 / 504 / 超时 → 等约 5s 后重试 |

业务重试用尽 → 如实标记该题并跳下一题，不在单题上空转。

## 调试 payload 分类（errorType）

- `compile` —— 编译错误（带编译器输出）
- `runtime` —— 运行时错误（如段错误）
- `wrong_answer` —— 自测输出与预期不符（带 实际 vs 预期）
- `test_error` —— 测试异常（页面没读到 output）
- `submit_failed` —— 提交评测未通过（带每条用例得分/内存/时间）

## 选下一题的顺序

从当前 active 题**之后**开始向后找第一个非 accepted 的题，找不到再从头绕回
（跳过当前题本身）。据此实现"跳过已 AC、按序推进"。

## 跨页面断点续做

跳题会重新加载页面，用会话级存储（sessionStorage）保持状态：

| key | 值 | 含义 |
|---|---|---|
| `pta-auto-submit` | `running` | 正在刷题 |
| `pta-auto-submit` | `solver-navigation` | 正在导航到下一题 |
| `pta-auto-submit-target` | `<problemId>` | 导航目标题 id |
| （无 key） | — | 用户未启动刷题 |

重新进入页面时读取该状态：若为 `running` / `solver-navigation` 则自动恢复续做；
若标记的目标题与当前 URL 不符（用户手动跳走了），则清除标记并停止。

## 等待类工具（SPA 异步渲染）

- 等编辑器挂载：轮询 `.cm-content[contenteditable="true"]` 出现，最长约 15s 兜底。
- 等题目状态变化：用 `MutationObserver` 观察列表 `<a>` 的 class 变化，
  目标题变为预期状态（如 accepted）即返回；进入观察前先同步查一次避免漏窗口；
  最长约 90s 兜底。
